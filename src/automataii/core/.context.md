# Context: src/automataii/core/

This sub-package, `core/`, within `automataii`, is intended to house the fundamental business logic, core data structures (models), and essential algorithms that form the backbone of the `automataii` application. It represents the heart of the application's domain-specific functionalities, independent of UI or specific utility implementations.

## Purpose:

- **Central Business Logic**: To implement the primary algorithms, computations, and state management related to the application's main purpose (e.g., animation processing, kinematic calculations, data analysis, simulation).
- **Data Modeling**: To define the core data structures and entities that the application works with. This is often done using Python classes, dataclasses, or Pydantic models.
- **Service Layer**: To provide a clear API for other parts of the application (like `gui` or `animate`) to interact with the core functionalities.

## Key Files & Potential Contents:

- **`models.py` / `models_pydantic.py` / `models_skeleton.py`**: These modules define the data structures used by the application.
    - `models_pydantic.py` strongly suggests the use of Pydantic for data validation, serialization, and settings management. These models would define the schema for objects like skeletons, parts, animation data, project settings, etc.
    - `models_skeleton.py` likely defines specific Pydantic or dataclass models related to skeletal structures.
    - `models.py` could be a more general-purpose model definition file or an older way of defining models if Pydantic is being progressively adopted.
- **`skeleton_manager.py`**: A manager class responsible for operations related to skeletons, such as creating, modifying, loading, saving, or processing skeletal data (bones, joints, hierarchy).
- **`project_data_manager.py`**: A manager class for handling overall project data. This could involve loading and saving project files, managing different components of a project (like animations, models, configurations), and providing access to this data.
- **`mechanism_manager.py`**: A manager class likely dealing with "mechanisms," which could refer to articulated systems, robotic components, or other complex assemblies that have defined motion or interaction.
- **`__init__.py`**: Makes `core/` a Python sub-package. It should export the key manager classes and core model classes that form the public API of this core package.
- **`__pycache__/`**: Standard Python bytecode cache (will be skipped).

## Interactions:

- **Consumed By**: `gui/`, `animate/`, `kinematics/`, command-line interfaces, or any other module that needs to perform or access core operations and data.
- **Uses**: May use general utilities from `src/automataii/utils/` and configurations from `src/automataii/config/`.

## Development Guidelines:

- **Separation from UI**: Core logic MUST remain independent of any GUI framework. The GUI should call into the core, not the other way around (unless using callbacks or signals for updates).
- **Clear API**: Define a clear, stable, and well-documented API for the core functionalities and managers.
- **Robustness and Error Handling**: Implement robust error handling, especially for operations involving data manipulation, file I/O, or complex computations.
- **Testability**: Core logic should be highly testable with unit tests. Manager classes should be designed to be testable, possibly by allowing injection of dependencies (like data loaders/savers).
- **Data Integrity**: If using Pydantic (as suggested by `models_pydantic.py`), leverage its validation capabilities to ensure data integrity.
- **Domain-Driven Design (DDD) Principles**: Consider applying DDD principles to model the core domain effectively, focusing on entities, value objects, aggregates, and services (managers can act as domain services).
